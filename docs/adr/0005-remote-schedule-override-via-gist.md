# Remote Schedule override via gist

The Schedule is compiled into the bundle (a macro slurps `seed.edn` at build), so changing it means a redeploy. To allow editing the Schedule without shipping, an optional localStorage pointer (`on-top/schedule-url`) may hold a raw-latest gist URL that is fetched at runtime. The gist holds EDN — the same shape as `seed.edn`, read data-only with `cljs.reader/read-string` — rather than JSON, which would force a string-key-to-keyword translation layer for no gain since the author already writes EDN. Fetching is stale-while-revalidate: paint instantly from a last-good cache (`on-top/schedule-cache`) or, failing that, the seed schedule floor, then revalidate in the background and swap in a fetched Schedule only if it parses to a map. A bad fetch (404, unparseable, offline) keeps what's showing and never poisons the cache; the failure is logged but invisible in the UI. This keeps the app instant and correct offline, which block-on-fetch and seed-first-then-swap both sacrifice. The seed stays compiled in as the permanent floor, so the app always renders something even with no pointer, no cache, and no network.
