# Per-surface gist Schedule overrides

The remote Schedule override from 0005 splits into two independent gists, one per surface, behind two config keys (`coreScheduleUrl` and `rareScheduleUrl`), superseding both the single `scheduleUrl` pointer and the combined `{:core … :rare …}` gist that 0008 had folded both surfaces into. The reason is how the author wants to manage the Schedules: editing one surface's routine without the other in front of them, two files that are reasoned about separately, matching the mental separation the two bounded contexts (0008) already have. A combined gist would put both routines in one file and one edit session, which is exactly the coupling the surfaces are meant to be free of. Everything else 0005 decided carries over unchanged and per surface: EDN over JSON, stale-while-revalidate from a per-surface cache key over the compiled-in seed floor, swap-in only on a clean parse, and a bad fetch that never poisons the cache and stays invisible in the UI. The same fetch and parser serve both surfaces, instantiated once each rather than written twice.
