# Per-surface completions, one table with a surface discriminator

Each surface keeps its own completion store — separate localStorage keys and separate outboxes — rather than pooling both into one map, because a Core Done-through and a Rare Done-through are coverage over different surfaces' occurrences and their task-ids share a namespace only by accident; a name reused across surfaces (nothing stops it) would silently clobber under a pooled store. They sync through the *same* mechanism and the *same* Supabase table from 0007, now keyed `(surface, task_id)` via one additive `surface` column on `completions(task_id text, done_through date)`. This keeps the config blob's single `completionsDbUrl` unchanged, writes the outbox-and-upsert sync code once and instantiates it per surface, and preserves every property of 0007 (forward-looking coverage, idempotent upserts, no delete path, storage bounded by task count). Chosen over a pooled single store, which would demand a never-violate `core/…` vs `rare/…` id-prefix convention and blur the two bounded contexts, and over two separate tables, which would put two endpoints and credentials in the config blob for no real isolation gain on a single-owner database.
